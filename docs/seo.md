# SEO & Discoverability

Search engine setup, meta tags, and strategic decisions on crawlability.

## Overview

Geetanjali is a React SPA with client-side rendering. This affects how search engines index content:

```
What crawlers see:
┌─────────────────────────────────────┐
│ <head>                              │
│   <!-- Static meta tags ✓ -->       │
│ </head>                             │
│ <body>                              │
│   <div id="root"></div>  ← Empty    │
│ </body>                             │
└─────────────────────────────────────┘
```

Google may execute JavaScript and index content, but with delays and inconsistency. We've optimized what we can without server-side rendering.

## Implementation

### Dynamic Meta Tags

The `useSEO` hook updates document head on each page:

```typescript
// frontend/src/hooks/useSEO.ts
useSEO({
  title: "Bhagavad Geeta 2.47",
  description: "Focus on action, not results...",
  canonical: "/verses/BG_2_47",
});
```

Updates: title, description, canonical, Open Graph, Twitter cards.

Used on all 14 pages. Falls back to defaults on unmount.

### Static Head (index.html)

```html
<!-- Primary -->
<title>Geetanjali - Ethical Guidance from the Bhagavad Geeta</title>
<meta name="description" content="..." />
<link rel="canonical" href="https://geetanjaliapp.com/" />

<!-- Open Graph -->
<meta property="og:title" content="..." />
<meta property="og:description" content="..." />
<meta property="og:image" content="https://geetanjaliapp.com/og-image.png" />

<!-- Twitter -->
<meta name="twitter:card" content="summary_large_image" />

<!-- JSON-LD -->
<script type="application/ld+json">
  { "@type": "WebApplication", ... }
</script>
```

### Sitemap

Dynamic XML sitemap generated by backend:

```
GET /sitemap.xml → 705 URLs (5 static + 700 verses)
```

- Cached in Redis (1 hour TTL)
- Includes lastmod, changefreq, priority
- Source: `backend/api/sitemap.py`

### RSS Feed

Daily verse feed for subscribers:

```
GET /feed.xml → Last 30 days of daily verses
```

- RSS 2.0 with Atom self-link
- Rich descriptions (paraphrase, translation, Sanskrit)
- Source: `backend/api/feed.py`

### robots.txt

```
User-agent: *
Allow: /
Disallow: /api/
Sitemap: https://geetanjaliapp.com/sitemap.xml
```

## Limitations

### Client-Side Rendering Gap

| Page | Crawlers See |
|------|--------------|
| Homepage | Correct meta tags (static in HTML) |
| Verse pages | Homepage defaults (JS not executed) |
| About, etc. | Homepage defaults |

The `useSEO` hook runs client-side. Crawlers that don't execute JavaScript see only the static `index.html` tags.

### Social Sharing

When users share verse pages on social media, previews show generic homepage content rather than verse-specific titles/descriptions.

## Strategic Decision: No Prerendering

We evaluated prerendering options and chose not to implement them.

### Options Considered

| Approach | Effort | Result |
|----------|--------|--------|
| vite-plugin-prerender | 4-6 hrs | Static pages only |
| vite-react-ssg | 8-12 hrs | All pages, medium maintenance |
| Next.js migration | 40+ hrs | Full SSR, high maintenance |

### Why We Skipped It

**Competitive landscape:** Verse content is available on established sites with 15-20+ years of backlinks (gitapress.org, holy-bhagavad-gita.org, iskcon.org). Ranking for "Bhagavad Gita verse 2.47" against these domains is unrealistic regardless of technical SEO.

**User intent mismatch:** Users searching for verse text want reference material. They're unlikely to convert to consultation users. Our value proposition (AI-powered guidance) isn't discoverable through verse searches.

**ROI assessment:** 8-16 hours of prerendering work for marginal SEO gains in a space we can't win doesn't justify the investment.

### When to Revisit

Consider prerendering if:
- Organic traffic becomes a strategic priority
- Blog/content marketing creates unique indexable content
- Domain authority grows through other channels
